# 基于 RFID 的小区车辆管理系统设计报告

## 一、课题名称

基于 RFID 的小区车辆管理系统设计

## 二、项目背景

### 2.1 项目发展趋势

随着物联网技术、无线通信技术和嵌入式技术的发展，传统小区车辆管理方式正在逐步向智能化、网络化和信息化方向转变。过去小区车辆出入多依赖人工登记、人工放行和纸质记录，这种方式存在效率低、数据不易保存、查询困难、管理不规范等问题。随着小区车辆数量增加，单纯依靠人工管理已经难以满足实际需求。

RFID 射频识别技术具有非接触式识别、响应速度快、成本较低、使用方便等特点，广泛应用于门禁、停车场、物流仓储、身份识别等场景。将 RFID 技术应用于小区车辆管理，可以通过卡片 UID 快速识别车辆身份，再结合数据库和后台管理系统实现车辆信息、车主信息和出入记录的统一管理。

同时，MQTT 协议作为一种轻量级物联网通信协议，适合嵌入式设备和云平台之间的数据传输。ESP32 主控板具有 WiFi 联网能力，能够方便地将 RFID 识别结果上传至云端或后台系统。通过 RFID、ESP32、MQTT、数据库和 Web 管理端的综合应用，可以构建一个较完整的物联网车辆管理系统。

### 2.2 项目研究意义

本项目以小区车辆出入管理为应用场景，设计并实现了一个基于 RFID 的小区车辆管理系统。系统能够完成车辆身份识别、后台权限判断、道闸模拟控制、出入记录保存和 Web 信息管理等功能。该设计不仅体现了感知识别、网络通信、后台管理和综合应用之间的联系，也能够较好地展示物联网系统的分层结构和实际应用价值。

通过本项目的设计，可以加深对物联网系统架构、嵌入式设备通信、数据库管理和 Web 应用开发的理解，具有较强的课程设计实践意义。

## 三、项目功能分析

根据课题要求和小区车辆管理的实际需求，本系统主要实现以下功能。

### 3.1 车辆身份识别功能

系统通过 RFID 卡片作为车辆身份凭证。每辆车绑定一张 RFID 卡片，卡片 UID 与车辆信息、车主信息建立对应关系。当车辆进出小区时，通过读卡器读取 UID，实现车辆身份识别。

### 3.2 车辆权限判断功能

后台系统根据 RFID UID 查询数据库，判断车辆是否允许通行。判断条件包括：

1. 卡片是否已经登记；
2. 车辆状态是否正常；
3. 卡片是否挂失；
4. 月租或通行有效期是否过期。

系统能够区分正常车辆、挂失车辆、过期车辆和未登记车辆，并给出不同处理结果。

### 3.3 道闸控制功能

当后台判断车辆允许通行时，硬件端控制舵机转动，模拟小区道闸打开；保持一段时间后，舵机复位，模拟道闸关闭。当车辆被拒绝通行时，道闸保持关闭状态。

### 3.4 信息显示功能

硬件端通过 OLED 显示屏显示系统运行状态，例如在线等待刷卡、正在验证、允许通行、拒绝通行、卡片过期、卡片挂失等信息，使系统运行过程更加直观。

### 3.5 车主信息管理功能

Web 管理端提供车主信息的查询、添加、修改和删除功能。车主信息包括车主姓名、联系电话、住址和备注等内容。

### 3.6 车辆信息管理功能

Web 管理端提供车辆信息管理功能，包括车牌号、所属车主、RFID 卡 UID、车辆类型、车辆状态和到期日期等。管理员可以新增车辆、修改车辆信息、设置车辆挂失状态或修改到期时间。

### 3.7 出入信息管理功能

每次刷卡后，后台自动保存一条出入记录。记录内容包括刷卡时间、卡号、车牌号、车主姓名、通行结果和原因。管理员可以通过 Web 页面查询、添加、修改和删除出入信息。

### 3.8 系统设置功能

系统提供管理员登录和密码修改功能。管理员登录后才可以进入后台进行数据管理，保证系统具有基本的访问控制能力。

### 3.9 综合查询功能

系统提供综合查询入口，管理员可以根据关键词查询车主信息、车辆信息和出入记录，方便快速定位相关数据。

## 四、物联网四层架构设计

根据物联网系统的一般结构，本项目可划分为感知识别层、网络构建层、管理服务层和综合应用层。四层之间相互配合，共同完成车辆管理系统的整体功能。

```text
┌────────────────────────────────────────────┐
│ 综合应用层                                  │
│ Web 管理端：登录、查询、车主管理、车辆管理、出入记录管理 │
└───────────────────▲────────────────────────┘
                    │ HTTP / 页面交互
┌───────────────────┴────────────────────────┐
│ 管理服务层                                  │
│ Flask 后台服务、SQLite 数据库、业务判断、记录保存        │
└───────────────────▲────────────────────────┘
                    │ MQTT 消息处理
┌───────────────────┴────────────────────────┐
│ 网络构建层                                  │
│ ESP32 WiFi、MQTT 协议、EMQX 云平台、主题订阅与发布      │
└───────────────────▲────────────────────────┘
                    │ 设备数据上传 / 控制指令下发
┌───────────────────┴────────────────────────┐
│ 感知识别层                                  │
│ RFID 卡片、RC522 读卡器、OLED 显示、SG90 舵机          │
└────────────────────────────────────────────┘
```

### 4.1 感知识别层

感知识别层是物联网系统的数据来源和执行基础。本系统的感知识别层主要由 RFID 卡片、RC522 读卡器、OLED 显示屏和 SG90 舵机组成。其中 RFID 卡片用于表示车辆身份，RC522 读卡器负责读取卡片 UID，OLED 显示屏用于显示系统状态，SG90 舵机用于模拟道闸动作。

该层的核心作用是完成车辆身份采集和现场反馈。若没有感知识别层，系统无法获取车辆身份，也无法执行通行控制。

### 4.2 网络构建层

网络构建层负责将感知识别层采集的数据传输到后台，并将后台的控制结果返回给硬件设备。本系统中，ESP32 通过 WiFi 接入网络，使用 MQTT 协议与 EMQX 云平台通信。

系统设计了两个 MQTT 主题：

| 主题 | 方向 | 功能 |
|------|------|------|
| `community/gate/request` | 硬件端 → 后台端 | 上传 RFID 刷卡请求 |
| `community/gate/response` | 后台端 → 硬件端 | 下发通行判断结果 |

网络构建层实现了设备端与后台端之间的解耦，使硬件端和软件端可以通过云平台完成异步通信。

### 4.3 管理服务层

管理服务层是系统的业务处理中心。本系统的管理服务层由 Python Flask 后台、MQTT 监听线程和 SQLite 数据库组成。该层负责接收硬件端上传的 UID，查询数据库，判断车辆状态，并将判断结果返回给硬件端。同时，该层还负责保存出入记录和维护车辆、车主等基础数据。

管理服务层的主要功能包括：

1. 数据库初始化与数据持久化；
2. 车辆通行权限判断；
3. 出入记录自动保存；
4. 管理员登录验证；
5. 为 Web 页面提供数据查询和管理服务。

### 4.4 综合应用层

综合应用层面向最终用户，即小区车辆管理员。本系统通过浏览器访问 Web 管理端，实现车主信息管理、车辆信息管理、出入信息管理、综合查询、密码修改和首页统计展示等功能。

综合应用层将底层感知数据和后台业务结果以可视化方式呈现给管理员，使系统具备良好的人机交互能力。

### 5.2 系统业务流程

```text

 ↓
系统初始化
 ↓
硬件端等待刷卡
 ↓

 ↓

 ↓
后台接收 UID 并查询数据库
 ↓
判断车辆是否登记
 ├─ 否 → 返回 DENIED，记录“未登记车辆”
 └─ 是
     ↓
   判断车辆是否挂失
     ├─ 是 → 返回 DENIED，记录“卡片挂失”
     └─ 否
         ↓
       判断车辆是否过期

         └─ 否 → 返回 APPROVED，记录“正常通行”
 ↓

 ↓
OLED 显示结果
 ↓
允许则舵机开闸，拒绝则保持关闭
 ↓
结束本次刷卡流程
```

## 六、成员分工与模块设计

本项目由两名成员共同完成，采用软硬件分工合作方式。

| 成员 | 负责模块 | 主要任务 |
|------|----------|----------|
| 成员一 | 硬件端设计 | ESP32、RFID、OLED、舵机、WiFi、MQTT 设备端程序设计与调试 |
| 成员二 | Web 端软件设计 | Flask 后台、SQLite 数据库、Web 管理页面、业务逻辑和数据持久化设计 |

### 6.1 成员一：硬件端模块设计

#### 6.1.1 硬件端设计目标

硬件端主要负责车辆身份采集和现场控制。其设计目标是能够稳定读取 RFID 卡片 UID，将数据通过网络上传给后台，并根据后台返回结果完成 OLED 显示和舵机控制。

#### 6.1.2 硬件端模块组成

```text
┌──────────────┐
│ RFID 卡片     │
└──────┬───────┘
       ↓
┌──────────────┐
│ RC522 读卡器  │
└──────┬───────┘
       ↓
┌─────────────────────────────┐
│ ESP32 主控                   │
│ WiFi 连接 / MQTT 通信 / 数据封装 │
└──────┬───────────────┬──────┘
       ↓               ↓
┌──────────────┐   ┌──────────────┐
│ OLED 显示屏   │   │ SG90 舵机     │
│ 显示验证结果   │   │ 模拟道闸开关   │
└──────────────┘   └──────────────┘
```

#### 6.1.3 硬件端流程设计

```text
硬件上电
 ↓
初始化 WiFi、MQTT、RFID、OLED、舵机
 ↓
显示 ONLINE 和 Swipe Card
 ↓
检测 RFID 卡片
 ↓
读取并格式化 UID
 ↓
显示 CHECKING
 ↓
发布 MQTT 消息到 request 主题
 ↓
等待 response 主题返回结果

DENIED：显示 DENIED，舵机保持关闭

```

#### 6.1.4 硬件端设计说明

硬件端以 ESP32 为核心。ESP32 具有 WiFi 联网能力，适合承担物联网边缘设备角色。RC522 负责 RFID 卡片识别，OLED 用于现场状态反馈，SG90 舵机用于模拟小区道闸开关。硬件端不直接保存车辆信息，而是将读取到的 UID 发送给后台，由后台统一判断，这样可以保证数据集中管理，便于管理员在 Web 端修改车辆状态后立即生效。

### 6.2 成员二：Web 端软件模块设计

#### 6.2.1 Web 端设计目标

Web 端软件主要负责系统管理、业务判断和数据持久化。其设计目标是提供一个可视化管理平台，使管理员能够方便地维护车主、车辆和出入记录，并能与硬件端完成实时联动。

#### 6.2.3 Web 端功能设计

Web 端包括以下页面和功能：

1. 登录页面：管理员输入账号和密码登录系统；
2. 系统首页：展示车辆数量、车主数量、今日刷卡次数和最近出入记录；
3. 车主信息管理：实现车主信息的查询、添加、修改和删除；
4. 车辆信息管理：实现车辆信息的查询、添加、修改和删除，并支持车辆状态设置；
5. 出入信息管理：记录每次刷卡事件，支持条件查询和管理；
6. 信息查询：根据关键词综合查询车辆、车主和出入信息；
7. 系统设置：管理员可修改登录密码。

#### 6.2.4 Web 端业务判断流程

```text

 ↓

 ↓

```

Web 端使用 SQLite 数据库，主要包括四张表：

| Owners | 保存车主基础信息 |

| AccessLogs | 保存每次刷卡出入记录 |

## 七、关键技术设计

### 7.1 RFID 识别设计

系统采用 RFID 卡片作为车辆身份标识。每张卡片具有唯一 UID，后台将 UID 与车辆信息绑定。当车辆刷卡时，硬件端只负责读取 UID，后台负责权限判断。这种设计降低了硬件端复杂度，也使车辆信息可以在 Web 端集中维护。

### 7.2 MQTT 通信设计

系统采用 MQTT 发布/订阅模式实现硬件端与后台端通信。ESP32 发布刷卡请求，后台订阅请求主题；后台发布判断结果，ESP32 订阅响应主题。该方式适合物联网场景，具有轻量、实时、低耦合等特点。

### 7.3 数据持久化设计

系统采用 SQLite 数据库保存管理员、车主、车辆和出入记录。SQLite 无需单独部署数据库服务，适合课程设计和本地演示。每次刷卡结果都会写入 AccessLogs 表，保证出入信息可追溯。

### 7.4 Web 可视化管理设计

Web 管理端采用 Flask 和 HTML/CSS/JavaScript 实现。管理员通过浏览器即可完成系统管理操作，不需要直接操作数据库。页面采用后台管理系统布局，包含侧边栏菜单、顶部状态栏、表格管理页面和表单页面，提高了系统可用性。

## 八、系统测试与结果分析

### 8.1 测试场景设计

系统测试覆盖以下典型业务场景：

| 测试场景 | 测试内容 | 预期结果 |
|----------|----------|----------|
| 正常车辆刷卡 | 已登记、状态正常、未过期 | 允许通行，舵机开闸 |
| 挂失车辆刷卡 | 已登记但状态为挂失 | 拒绝通行，舵机不开闸 |
| 过期车辆刷卡 | 已登记但到期时间已过 | 拒绝通行，舵机不开闸 |
| 未登记卡片刷卡 | 数据库中无该 UID | 拒绝通行，舵机不开闸 |
| Web 端管理 | 添加、修改、删除车辆和车主 | 数据库正常更新 |
| 出入记录查询 | 按关键词和结果查询 | 能正确筛选记录 |

### 8.2 测试结果

经过联调测试，系统能够完成以下功能：

1. ESP32 能够连接 WiFi 并接入 MQTT 云平台；
2. RFID 读卡器能够读取实体卡片 UID；
3. 后台能够接收刷卡请求并查询数据库；
4. 系统能够正确区分正常、挂失、过期和未登记车辆；
5. 判断结果能够通过 MQTT 返回至硬件端；
6. OLED 能够显示通行结果；
7. 舵机能够在允许通行时模拟开闸；
8. Web 端能够完成车主、车辆和出入记录管理；
9. 每次刷卡记录能够自动保存，便于后续查询。

测试结果表明，本系统实现了预期功能，物联网四层结构完整，各层之间能够协同工作。

## 九、项目总结

本项目完成了一个基于 RFID 的小区车辆管理系统设计与实现。系统从物联网四层架构出发，将感知识别层、网络构建层、管理服务层和综合应用层进行了较完整的设计。硬件端负责 RFID 识别、现场显示和道闸控制；网络层通过 WiFi 和 MQTT 实现设备与后台之间的数据通信；管理服务层完成数据库管理、通行判断和记录保存；综合应用层通过 Web 页面为管理员提供可视化管理功能。

项目由两名成员分工完成，一名成员负责硬件端设计与调试，另一名成员负责 Web 端软件设计与实现。通过分工协作，系统完成了从刷卡识别到后台判断，再到硬件执行和数据记录的完整闭环。

本系统能够较好地模拟小区车辆出入管理场景，具有一定的实用价值。后续如果继续完善，可以增加访客车辆登记、车牌识别、数据导出、移动端访问、云服务器部署等功能，使系统更加接近实际工程应用。
