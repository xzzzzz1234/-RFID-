# 基于RFID的小区车辆管理系统

## 项目简介

本系统是一个面向小区车辆通行场景的物联网管理平台，基于 RFID 射频识别技术实现车辆自动识别与门禁控制。系统由硬件端（ESP32 + RFID + 舵机 + OLED）和软件端（Flask Web 管理后台 + MQTT 通信）两部分组成，实现了从刷卡识别到云端判断再到物理开闸的完整闭环。

## 设计背景

随着城市化进程加快，小区车辆管理面临以下挑战：

- 传统人工登记效率低、易出错
- 纸质记录难以查询和统计
- 外来车辆难以有效管控
- 月租车辆到期难以及时提醒

本系统通过 RFID 技术实现车辆自动识别，结合物联网云平台实现远程管理，有效解决上述问题。

根据课设要求，系统实现以下功能模块：

| 序号 | 模块 | 功能说明 |
|:----:|------|----------|

| 3 | 车主信息管理 | 车主的查询、添加、修改、删除 |
| 4 | 车辆信息管理 | 车辆的查询、添加、修改、删除，支持状态和有效期管理 |

| 6 | 系统首页 | 数据统计仪表盘、最近出入记录、MQTT 连接状态监控 |

## 系统特色

- **软硬件联动**：刷卡即触发完整业务流程，无需人工干预
- **云端通信**：基于 MQTT 协议，支持远程管理和实时响应
- **多状态管理**：支持正常通行、卡片挂失、卡片过期、未登记车辆四种状态
- **记录可追溯**：每次刷卡自动保存出入记录，支持多维度查询
- **现代化界面**：响应式 Web 设计，支持多端访问

## 硬件系统设计

### 硬件总体组成

硬件端以 ESP32 开发板为主控，通过 RFID 读卡器采集卡片 UID，通过 WiFi 接入 MQTT 云平台，并根据后台返回结果控制 OLED 显示、舵机开闸和提示音反馈。

```
┌──────────────┐
│  RFID 卡片    │
└──────┬───────┘
       │ 13.56MHz 感应
┌──────▼───────┐
│ RC522 读卡器  │
└──────┬───────┘
       │ SPI
┌──────▼────────────────────────────┐
│              ESP32 主控             │
│  读卡 / WiFi / MQTT / OLED / 舵机控制 │
└──────┬─────────────┬───────────────┘
       │ I2C         │ PWM
┌──────▼───────┐ ┌───▼────────┐
│ SH1106 OLED   │ │ SG90 舵机   │
│ 显示通行结果   │ │ 模拟道闸     │
└──────────────┘ └────────────┘
```

| 序号 | 器件 | 型号 | 数量 | 作用 |
|:----:|------|------|:----:|------|
| 1 | 主控开发板 | ESP32 / ESP32-S3 | 1 | 负责联网、读卡、通信和外设控制 |

| 5 | 舵机 | SG90 | 1 | 模拟小区道闸开关 |
| 6 | 音频模块 | MAX98357A I2S 功放（可选） | 1 | 播放通行或拒绝提示音 |

### 引脚连接说明

#### RC522 RFID 模块（SPI）

| RC522 引脚 | ESP32 引脚 | 说明 |
|-----------|------------|------|
| SDA / SS | GPIO 5 | SPI 片选 |
| SCK | GPIO 18 | SPI 时钟 |
| MOSI | GPIO 23 | SPI 主发从收 |
| MISO | GPIO 19 | SPI 主收从发 |
| RST | GPIO 4 | 复位引脚，避让 OLED I2C 引脚 |
| 3.3V | 3.3V | 电源 |
| GND | GND | 地 |

#### OLED 显示屏（I2C）

| OLED 引脚 | ESP32 引脚 | 说明 |
|----------|------------|------|
| SDA | GPIO 21 | I2C 数据线 |
| SCL / SCK | GPIO 22 | I2C 时钟线 |
| VCC | 3.3V / 5V | 电源，按模块标识连接 |
| GND | GND | 地 |

> 说明：项目中曾遇到 1.3 寸 OLED 花屏问题，最终将屏幕驱动适配为 SH1106，并使用 ESP32 默认 I2C 引脚 SDA=21、SCL=22，避免与 RFID 的 SPI/RST 引脚冲突。

#### SG90 舵机

| 舵机线 | ESP32/电源 | 说明 |
|--------|------------|------|
| 信号线 | GPIO 13 | PWM 控制信号 |
| VCC | 5V | 舵机供电 |
| GND | GND | 共地 |

> 注意：舵机工作电流较大，如出现 ESP32 重启、舵机抖动等情况，建议舵机使用独立 5V 电源，并确保 ESP32 与舵机电源共地。

#### MAX98357A I2S 音频模块（可选）

| 模块引脚 | ESP32 引脚 | 说明 |
|----------|------------|------|
| BCLK | GPIO 14 | I2S 位时钟 |
| LRC | GPIO 27 | I2S 左右声道时钟 |
| DIN | GPIO 16 | I2S 数据输入 |
| VIN | 5V / 3.3V | 电源 |
| GND | GND | 地 |

### 硬件端代码说明

| 目录 | 文件 | 作用 |
|------|------|------|
| `hardware/wifi/` | `wifi.ino` | WiFi 连接测试，验证 ESP32 能否接入本地网络 |
| `hardware/scan/` | `scan.ino` | I2C 扫描测试，用于确认 OLED 屏幕地址和引脚连接 |
| `hardware/duka/` | `duka.ino` | RFID 读卡测试，用于读取卡片 UID 和确认 RC522 工作正常 |
| `hardware/duoji/` | `duoji.ino` | 项目硬件主程序，集成 WiFi、MQTT、RFID、OLED、舵机和提示音 |

> 注：当前项目中 `duoji.ino` 是已集成的主程序，`duka.ino` 为 RFID 读卡测试程序。文件名沿用了调试阶段命名。

### 硬件端主程序流程

硬件主程序主要完成以下任务：

1. 初始化串口、OLED、I2S 音频、WiFi、MQTT、SPI、RC522 和舵机；
2. OLED 显示 `ONLINE`，提示等待刷卡；
3. 检测到 RFID 卡片后读取 UID；
4. 将 UID 封装为 JSON 数据；
5. 通过 MQTT 发布到 `community/gate/request`；
6. 等待后台通过 `community/gate/response` 返回判断结果；
7. 如果结果为 `APPROVED`，OLED 显示 `PASS`，播放提示音，舵机开闸 3 秒后复位；
8. 如果结果为 `DENIED`，OLED 显示 `DENIED`，根据原因显示过期、挂失或无效卡，并保持闸门关闭。

### 硬件端 MQTT 数据格式

ESP32 上传刷卡请求：

```json
{
  "device_id": "ESP32_Gate_01",
  "card_uid": "F1CD2203"
}
```

后台返回通行结果：

```json
{
  "status": "APPROVED",
  "owner": "张三",
  "plate": "浙A·88888",
  "msg": "Welcome Home"
}
```

### 硬件调试步骤

建议按以下顺序逐步调试，避免一次集成导致问题难以定位：

1. **WiFi 调试**：运行 `hardware/wifi/wifi.ino`，确认串口输出 `WiFi connected!` 和 IP 地址；
2. **OLED 调试**：运行 `hardware/scan/scan.ino`，确认 I2C 地址，当前项目使用地址 `0x3C`；
3. **RFID 调试**：运行 `hardware/duka/duka.ino`，刷卡后串口应输出卡片 UID；
4. **舵机调试**：确认 GPIO 13 输出 PWM 后舵机可以开关；
5. **MQTT 调试**：确认 ESP32 成功连接 `broker.emqx.io`，并订阅 `community/gate/response`；
6. **联调测试**：启动软件端 `python app.py`，刷测试卡片，观察网页记录、OLED 显示和舵机动作。

### 已解决的硬件问题

| 问题 | 原因 | 解决方式 |
|------|------|----------|
| OLED 花屏 | 1.3 寸 OLED 实际驱动与 SSD1306 不完全一致 | 使用 SH1106/兼容驱动思路，并固定默认 I2C 引脚 21/22 |
| OLED 与 RFID 引脚冲突 | RFID RST 与 I2C 引脚规划冲突 | 将 RFID RST 调整到 GPIO 4 |
| 卡片 UID 格式不统一 | 串口读出可能带空格或大小写不一致 | 软件端统一去空格并转大写 |
| 舵机动作不稳定 | 舵机供电电流不足 | 建议外接 5V 电源并共地 |
| MQTT 异步响应 | 刷卡请求和后台判断不在同一设备 | 使用 `request/response` 两个主题实现双向通信 |

## 常见问题

### Q: MQTT 状态显示「未连接」怎么办？

A: 检查以下几点：
1. 确保电脑已联网
2. 确保 `broker.emqx.io` 可访问（可尝试 ping）
3. 重启 `app.py` 程序

### Q: 刷卡后网页没有新记录？

A: 检查以下几点：
1. 确保硬件端 WiFi 已连接
2. 确保硬件端 MQTT 已连接
3. 确保软件端 MQTT 状态为「已连接」
4. 刷新网页查看

### Q: 如何添加新的 RFID 卡片？

A: 操作步骤：
1. 先在「车主信息管理」中添加车主
2. 然后在「车辆信息管理」中添加车辆
3. 填写车牌号、选择车主、输入新卡片的 UID
4. 设置状态为「正常」，填写到期日期

### Q: 如何获取新卡片的 UID？

A: 使用 `hardware/scan/scan.ino` 程序：
1. 上传到 ESP32
2. 打开串口监视器
3. 刷卡后会显示 UID

### Q: 数据库在哪里？如何备份？

A: 数据库文件为 `software/gate_system.db`，直接复制该文件即可备份。
